# 關卡產生器理論筆記

來源：`桃園市立平興國中-作品說明書-65-國中組-數學科-感染圖騰-複賽.docx`，以及使用者補充的結論截圖。

這份筆記只記錄對關卡產生器有用的結論。原始研究是 2-鄰居 bootstrap percolation，規則與目前 `src/domain/engine.ts` 的感染規則一致。

## 基本定義

- 網格：由單位方塊以邊相接形成的連通圖形；只碰到頂點不算相連。
- 鄰居：共用邊的方塊。只看上下左右，不看斜角。
- 孤立方塊：在圖形內只有 1 個鄰居的方塊。
- 感染源：一開始就被選為感染方塊的格子。
- 感染規則：未感染方塊若至少有 2 個已感染鄰居，就會在下一波感染，之後也具有感染能力。
- 完全感染：最後所有非空洞格都被感染。
- 完全感染方塊數：能使整個網格完全感染的最少初始感染源數。對遊戲來說，這就是該關卡的數學最佳解種子數。

## 對關卡資料的意義

- `holes` 對應不屬於網格的空格。它們不需要被感染，也不能提供鄰居數。
- `requiredSeeds` 只適合用在關卡想固定某一部分最佳構造時。每個必選點都會提高產生器的強制條件。
- `blockedSeeds` 仍然必須被感染，但不能作為初始感染源。若所有最小解都必須用到某個 blocked 格，該關卡應被產生器拒絕。
- 孤立方塊一定要是感染源，因為它永遠不可能在後續取得 2 個已感染鄰居。
- 細長尾巴、一格寬附肢、L 形尾端常出現「從末端開始，每隔一格放一個感染源」的最佳構造。
- 星等門檻應從完全感染方塊數推導：
  - `stars.three` 通常應等於最少感染源數。
  - `stars.two`、`stars.one` 可作為較寬鬆的過關容許值。
  - `maxSeeds` 仍維持目前 app 的定義：最低過關門檻，不一定等於數學最佳解，除非該關只有 3 星/失敗兩種結果。

## 周長下界

原文給出重要下界：

- 對任意連通網格，若周長為 `P`，完全感染方塊數至少是 `ceil(P / 4)`。
- 理由：感染擴張不會增加已感染區域的周長；若一開始感染源太少，最後不可能形成目標圖形的周長。

產生器用法：

- 先由產生出的 playable cells 計算周長 `P`。
- 若預設種子預算小於 `ceil(P / 4)`，可直接拒絕。
- 這只是下界，不是任意圖形的精確答案；實際關卡仍要用 solver 或枚舉驗證。

矩形網格是特例。原文引用前一屆研究結果：`m x n` 矩形的完全感染方塊數等於周長下界，也就是 `ceil((m + n) / 2)`。

## 可實作公式

以下公式來自結論截圖。文件中的符號保留原意，但在程式中應轉成明確參數名稱。

### 廣義十字形

廣義十字形：

```text
R(m x n) / S1({a_k}_{k=1}^p) ∪ S2({b_k}_{k=1}^q) ∪ S3({c_k}_{k=1}^r) ∪ S4({d_k}_{k=1}^s)
```

完全感染方塊數為：

```text
ceil((m + n) / 2)
```

或：

```text
ceil((m + n) / 2) + 1
```

產生器用法：

- 廣義十字形可當作「外框 `m x n` 矩形扣除四個階梯角落」的模板。
- `stars.three` 只能在 `ceil((m + n) / 2)` 與 `ceil((m + n) / 2) + 1` 之間選一個。
- 因為結論截圖沒有列出判斷加 1 的完整 case，實作時應由 solver 判定兩者哪個是正確最小值。

### 簍空形

簍空形：

```text
R(m x n) - X
```

其中 `X` 可以是：

- 矩形網格 `R(a x b)`；
- 或廣義十字形網格。

若內洞 `X` 的外框尺寸為 `a x b`，完全感染方塊數為：

```text
ceil((m + n + a + b) / 2)
```

產生器用法：

- 外框是 `m x n`。
- 內洞 `X` 不可碰到外框邊界，否則不符合簍空形定義。
- 即使 `X` 是廣義十字形，公式仍用內洞外框的 `a + b`。
- 程式資料可把內洞格全部轉成 `holes`，再檢查剩餘 playable cells 仍連通。

### 蜈蚣形

蜈蚣形：

```text
C(n; {a_k}_{k=1}^m; {b_k}_{k=1}^m; {c_k}_{k=1}^{m-1})
```

當 `n >= 2` 時，先定義數列 `{x_k}_{k=1}^{m-1}`。

首項：

```text
x_1 = (a_1 mod 2) + (a_2 mod 2) + (b_1 mod 2) + (b_2 mod 2)
```

其餘項，`k = 2, 3, ..., m - 1`：

```text
x_k = 0, if (a_{k+1} mod 2) + (b_{k+1} mod 2) = 1 or 2
x_k = 1, if (a_{k+1} mod 2) + (b_{k+1} mod 2) = 0
```

當 `x_1 = 0, 1, 2, 3` 時，完全感染方塊數為：

```text
ceil((c_1 + n + 2 - x_1) / 2)
+ sum_{k=2}^{m-1} ceil((c_k + x_k) / 2)
+ sum_{k=1}^{m} (ceil(a_k / 2) + ceil(b_k / 2))
```

當 `x_1 = 4` 時，完全感染方塊數為：

```text
ceil((c_1 + n + 3 - x_1) / 2)
+ sum_{k=2}^{m-1} ceil((c_k + x_k) / 2)
+ sum_{k=1}^{m} (ceil(a_k / 2) + ceil(b_k / 2))
```

產生器用法：

- `a_k`、`b_k` 可視為左右腳長度，`c_k` 可視為相鄰身體段之間的連接量，`n` 是軀幹厚度或基礎身體參數；實作前應先把專案中的 centipede generator 參數命名固定下來。
- 腳長的奇偶性會影響 `x_k`，因此蜈蚣形可用來做難度微調。
- 新增腳或延長腳通常會透過 `ceil(a_k / 2)`、`ceil(b_k / 2)` 直接增加最少感染源。

## 最少感染源計算流程

對任意程序產生的關卡，以現有感染引擎作為最終判定：

1. 從 `rows`、`cols`、`holes` 建立 playable cell 集合。
2. 先算快速限制：
   - 周長下界 `ceil(P / 4)`；
   - 所有 degree-1 格必須是感染源；
   - `requiredSeeds` 必須包含；
   - `blockedSeeds` 必須排除在初始候選外。
3. 若關卡屬於上述模板，先用對應公式得到候選最小值。
4. 從下界開始，按感染源數由小到大搜尋候選集合；若公式已給候選值，可優先驗證該值，再檢查是否存在更小解。
5. 對每個候選集合呼叫 `runInfection(level, seeds)`。
6. 第一個能完成感染的集合大小，就是完全感染方塊數。
7. 可保存一組或多組最小解，用於編輯器提示、QA 或自動測試。

效能建議：

- 小棋盤或強模板關卡可用完整枚舉求精確最小值。
- 大棋盤應先用形狀模板建構候選解，再用 solver 檢查是否存在更小解。
- 即使形狀來自原文已證明的類型，也要用目前 engine 驗證：指定感染源能完成關卡，且沒有更少的感染源能完成關卡。

## 實作注意

- `ceil` 在 TypeScript 中使用 `Math.ceil`。
- `mod 2` 在參數皆為正整數時可用 `value % 2`。
- 公式輸出的完全感染方塊數應同步到 `stars.three`。
- 若使用 `requiredSeeds` 或 `blockedSeeds` 改變玩家可選集合，必須重新跑 solver；原公式只描述原始圖形，不包含額外強制或禁止條件。
